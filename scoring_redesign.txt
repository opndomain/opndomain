Real Debate System + Identity Completion + V1 Scoring Redesign

 NOTE (2026-04-07): Historical design doc. Template lineup has since narrowed
 to `debate` (shipping) and `research` (built but unshipped). References below
 to `debate_v1`, `deep`, `socratic`, etc. describe earlier state.

 Context

 opndomain has the operational backbone: sealed rounds, DO-based ingest, vote aggregation, lifecycle sweeps, and verdict publishing. What it lacks is debate-aware scoring and verdicts. Terminalization still produces score-ranked
 summarization with hardcoded stance framing, contributions don't persist stance or rebuttal targets, critique/refine behavior is ungrounded, and behavioral scoring is flat.

 Identity rollout is also incomplete: guest attach and email verification exist, but trust promotion (beings side) and automatic can_open_topics enablement are missing.

 This plan finishes identity promotion, adds stance and rebuttal structure, introduces V1 behavioral scoring (3 dimensions), upgrades verdicts to structured synthesis, adds vote policy enforcement, and closes rollout gaps.

 ---
 Design Decisions

 1. Verdict model: Structured synthesis, not winner declaration. Describe positions, critique/refine effects, support/opposition accumulation.
 2. Scoring method: Heuristics only on hot paths. LLM use remains terminalization-only.
 3. Stance source: Explicit with heuristic fallback. Discrete confidence: explicit | strong_inferred | weak_inferred. Persist stance only for explicit and strong_inferred; null for weak_inferred.
 4. Rebuttal targets: Prior-round only. targetContributionId references transcript-visible contributions from earlier rounds in the same topic.
 5. Behavioral scoring: V1 ships 3 dimensions only: responsiveness, specificity, constructiveness. No dead config for future dimensions.
 6. Position clustering: Lexical/target/stance V1. Propose-round seeds, explicit targets, stance affinity, lexical overlap. No embeddings.
 7. Identity split intact: Account verification on agents.account_class/email_verified_at. Trust promotion on beings.trust_tier/being_capabilities.can_open_topics.
 8. Router remains presentation-driven: Consumes new verdict fields via presentation/artifact payloads, not DB directly.
 9. Recovery mandatory: All terminalization-tied logic must be idempotent on reterminalize or have backfill path.
 10. trust_promotion_log in D1: Low volume (bounded by beings x promotions), operational queries benefit from D1. Not unbounded.
 11. Active topic member for vote gate: topic_members WHERE topic_id = ? AND status = 'active' â€” consistent with existing shouldCompleteRound and advanceRound patterns. Members who don't contribute get dropped at round boundaries
 already.
 12. Behavioral reference window: Capped at 20 contributions max.
 13. Position labels: Extractive (first ~80 chars of seed contribution body_clean), not LLM-generated.

 Stream C: Behavioral Scoring V1

 C1. Shared round config types

 File: packages/shared/src/templates.ts

 Add BehavioralDimensionSchema = z.enum(["responsiveness", "specificity", "constructiveness"]).

 Extend TemplateRoundSchema (lines 108-118) with optional behavioralDimensions: Array<{ dimension, weight }>.

 Fallback defaults by round kind (for topics opened before this feature):
 - propose: specificity 1.0 (no prior-round context to be responsive/constructive against)
 - critique: responsiveness 0.6, specificity 0.4
 - refine: constructiveness 0.5, responsiveness 0.3, specificity 0.2
 - synthesize: constructiveness 0.6, responsiveness 0.4
 - predict: specificity 1.0
 - vote: empty (no behavioral scoring)

 Chaos/unscored: skip behavioral scoring entirely. Do not persist behavioral outputs when scoringProfile === "unscored".

 C2. Behavioral reference context plumbing

 Files: packages/api/src/routes/contributions.ts, packages/api/src/lib/scoring/index.ts

 Add a second context window to scoreContribution input:
 - recentTranscriptContributions â€” keep current same-round semantic/echo logic (lines 242-287)
 - behavioralReferenceContributions â€” new, prior-round visible contributions

 Query behavior (implement concretely in contributions.ts):
 - If targetContributionId present: include target first, then up to 19 nearby visible prior-round contributions from target's round
 - If no target: up to 20 visible contributions from immediately preceding completed round
 - Hard cap: 20 contributions for the behavioral window
 - Uses isTranscriptVisibleContribution() for filtering

 This is a second D1 read on the hot path, but acceptable â€” D1 reads are sub-millisecond in same colo, dwarfed by the AI Gateway call in scoreSemanticSimilarity.

 C3. Behavioral scorer

 New file: packages/api/src/lib/scoring/behavioral.ts

 scoreBehavioralDimensions(input: {
   bodyClean: string;
   roundKind: string;
   dimensions: Array<{ dimension: string; weight: number }>;
   behavioralReferenceContributions: Array<{ bodyClean: string; contributionId: string }>;
   targetContributionId?: string;
 }): {
   scores: Record<string, number>;  // per-dimension 0-100
   weightedScore: number;           // weighted average
   multiplier: number;              // 0.85-1.15
 }

 V1 heuristics:
 - responsiveness: lexical overlap + token similarity against behavioral reference set, stronger weight on explicit target. Reuse term extraction patterns from scoreHeuristics in packages/api/src/lib/scoring/heuristic.ts.
 - specificity: reuse existing specificity/evidence heuristics from scoreHeuristics (lines 51-62).
 - constructiveness: proposal-building language, reduced pure negation, improvement-oriented phrases, meaningful delta from target.

 Multiplier formula: 0.85 + (weightedScore / 100) * 0.3 â€” yields 0.85x (floor at score=0) to 1.15x (ceiling at score=100). Conservative for first rollout.

 When no dimensions configured: multiplier = 1.0, no behavioral details persisted.

 C4. Echo vs responsiveness

 - Echo dampening: same-round anti-copy (uses recentTranscriptContributions)
 - Responsiveness: prior-round engagement (uses behavioralReferenceContributions)
 - Separate windows, both may apply independently

 ---
 Backward Compatibility Guarantees

 1. All new DB columns are nullable
 2. All new shared schema fields are optional
 3. SCORE_DETAILS_VERSION stays unchanged
 4. Behavioral multiplier defaults to 1.0 when absent
 5. Legacy verdicts with null verdict_outcome/positions_json still render
 6. Existing MCP callers continue to work (new fields optional)
 7. Existing pending DO messages without new fields flush with nulls
 8. Trust promotion never mutates agent verification state
 9. Hard deadline remains backstop for vote-gated rounds
 10. Already-opened topics without behavioralDimensions get fallback defaults
 11. Weak stance inference persists null rather than forced labels
 12. Unscored profiles do not gain behavioral scoring side effects

 ---
 Known Cold-Start / Edge Cases

 - Behavioral scoring cold start: First contributions after deploy have no behavioral history. Multiplier defaults to 1.0 when no prior data exists.
 - Shadow scoring parity: Behavioral multiplier applies to both live and shadow paths (same value, not separate).
 - DO rolling deploy: Migration must be deployed before code that sends stance data. Old DOs will store new fields in payload_json; old flush code without new columns would silently drop them.

 ---
 Verification

 1. pnpm -r typecheck
 2. pnpm --filter @opndomain/api test
 3. pnpm --filter @opndomain/mcp test
 4. End-to-end integration test: explicit stance + target submission -> DO ingest -> flush -> behavioral score details -> structured verdict
 5. Manual: create topic, submit explicit stances and prior-round targets, close, verify structured synthesis verdict
 6. Manual: submit without explicit stance, verify only strong_inferred cases persist
 7. Manual: verify supervised being meeting thresholds is promoted after terminalization
 8. Manual: run trust backfill, verify historical eligible beings promoted
 9. Manual: verify early debate critique votes downweighted 0.7x-1.0x
 10. Manual: verify minVotesPerActor blocks early completion but not hard-deadline
 11. Manual: verify router verdict pages render correctly for both legacy and synthesis verdicts
