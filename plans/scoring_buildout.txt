Plan: Port Claim Graph + Epistemic Engine (without LLM extraction)

 Context

 The authority docs (IDEAS-BANK.md, LLM_Claim_Extraction.txt, WORKING-TRUTH.md) define a claim graph and epistemic
 engine as a "Credible Later-Layer System." The legacy codebase at D:/moltzdev has working implementations. The current
  rebuild has zero claim/epistemic infrastructure. This plan ports the deterministic pieces — heuristic claim
 extraction, claim graph linking, epistemic engine resolution, and reputation feedback — without the LLM extraction
 path.

 This is a selective port with rebuild-specific rewrites, not a verbatim copy. Six structural issues from the legacy
 code are addressed explicitly.

 ---
 Structural Issues Addressed

 1. Reterminalization idempotency bug

 Problem: rebuildDomainReputation() wipes prior epistemic deltas, but claim_reputation_applied flags prevent
 reapplication.
 Fix: Add rebuildEpistemicAdjustmentsForTopic(db, topicId) — a deterministic function that recalculates all epistemic
 deltas from resolved claims and reapplies them fresh. Called on reterminalize path after rebuildDomainReputation().
 Ignores the claim_reputation_applied flag entirely — it resets all flags to 0, recalculates deltas from current
 resolution state, applies them, then sets flags to 1.

 2. Prediction table dependencies throughout engine

 Problem: Legacy references being_domain_prediction_reliability, being_prediction_reliability, and
 arena_prediction_scores in 3 functions, not just getPredictionSignal().
 Fix: Remove all three prediction queries entirely for this pass:
 - getPredictionSignal() → returns null always (already handles gracefully)
 - rebuildEpistemicReliabilityForBeingInDomain() lines 760-765 → predictiveReliability = 50 (neutral default),
 predictionEvidenceConfidence = 0
 - computeTopicEpistemicEvaluation() lines 968-971 → prediction_realization_rate = 50 (neutral default, no query)

 Each removal gets a // STUB: wire when prediction scoring is ported comment.

 3. Round-kind assumptions in evaluation metrics

 Problem: Legacy hardcodes round_type = 'propose', 'critique', 'synthesize' which don't cover all rebuild round kinds
 (e.g., predict, refine, vote, default).
 Fix: Scope initial port to round-agnostic metrics only. Three metric rewrites:
 - best_initial_claim_survival → use sequence_index = 0 (first round) instead of round_type = 'propose' AND
 round_number = 1
 - final_synthesis_claim_survival → use the last completed round (MAX(sequence_index) WHERE status = 'completed')
 instead of round_type = 'synthesize'
 - critique_precision_score → apply to any round where the contribution's detected role is 'critique' (from
 contribution_scores.details_json) rather than filtering on round_kind = 'critique'. For v1 simplification: apply to
 all contributions equally (score 50 default), with a // TODO: refine with role-aware critique detection comment.

 4. Migration registration/generation plumbing

 Problem: Adding 006_epistemic_core.sql alone doesn't register it. The rebuild requires: schema.generated.ts
 regeneration, schema.ts import + API_MIGRATIONS entry, and schema.test.ts assertions.
 Fix: Phase 1 includes all four files:
 - Create 006_epistemic_core.sql
 - Add EPISTEMIC_CORE_SQL entry to generate-schema-module.mjs
 - Run generator to update schema.generated.ts
 - Add import + migration entry in schema.ts
 - Add test case in schema.test.ts

 5. Claim flush failure blocking terminalization

 Problem: All unflushed pending_aux rows (any table_name) count toward remainingCount in flushPendingTopicState(). A
 claim flush failure or autoStakeClaimsForContribution() throw would block force-flush from draining, preventing topic
 closure.
 Fix: Two changes:
 - Claim flush is idempotent: INSERT OR IGNORE into contribution_claims, mark pending_aux flushed immediately on D1
 write success. No throw on duplicate.
 - Staking is decoupled: autoStakeClaimsForContribution() runs as best-effort after flush succeeds. Wrapped in
 try/catch — failure is logged but does not affect flushed status or remaining count. Staking can be retried later
 during terminalization (the epistemic engine orchestrator already calls stake settlement).

 6. Epistemic engine as hard terminalization dependency

 Problem: If evaluateEpistemicEngineForTopic() throws, topic closure fails entirely. Too aggressive for a later-layer
 system.
 Fix: Epistemic evaluation runs fail-open behind the flag:
 if (env.ENABLE_EPISTEMIC_SCORING) {
   try {
     epistemicResult = await evaluateEpistemicEngineForTopic(env.DB, topicId);
     await applyClaimResolutionToReputation(env.DB, topicId);
   } catch (error) {
     console.error(`epistemic evaluation failed for topic ${topicId}, continuing terminalization`, error);
   }
 }
 Verdict reasoning_json includes epistemic metrics only when available. Terminalization never fails due to epistemics.

 Create packages/api/src/db/006_epistemic_core.sql

 9 tables ported from legacy migration-no-human-epistemics.sql with rebuild naming:

 1. contribution_claims — extracted claims per contribution. Key columns: id, contribution_id, topic_id, claim_index,
 claim_text, verifiability_class, resolution_status, resolution_confidence, resolution_method, epistemic_weight,

 4. claim_resolution_evidence — signal records (evidence_type, signal_value, signal_confidence, payload_json).
 5. domain_claim_graph — cross-topic claim relationships (claim_id, related_claim_id, relationship, confidence,
 source_mechanism, domain_id). UNIQUE(domain_id, claim_id, related_claim_id, relationship).
 6. being_epistemic_reliability — per-being per-domain profile (counters + scores). UNIQUE(being_id, domain_id).
 7. reputation_stakes — claim outcome stakes. UNIQUE(claim_id, being_id).
 8. claim_challenges — direct challenges. UNIQUE(claim_id, challenger_being_id).
 9. topic_epistemic_evaluations — topic-level metrics. UNIQUE(topic_id).

 Plus indexes and updated_at triggers following 002_phase2_integrity.sql pattern.

 Phase 5: Epistemic Engine (Selective Port)

 Create packages/api/src/lib/epistemic/engine.ts

 Selective port of D:/moltzdev/packages/api/src/lib/epistemic-engine.ts with three categories of changes.

 A. SQL renames throughout

 All legacy table/column names -> rebuild names (see translation table above).

 B. Prediction dependencies removed (Issue #2)

 Three explicit removals:
 - getPredictionSignal() → always returns null
 - rebuildEpistemicReliabilityForBeingInDomain() → hardcoded predictiveReliability = 50, predictionEvidenceConfidence =
  0
 - computeTopicEpistemicEvaluation() → hardcoded prediction_realization_rate = 50

 Each gets // STUB: wire when prediction scoring is ported.

 C. Round-kind metrics made agnostic (Issue #3)

 - best_initial_claim_survival → WHERE r.sequence_index = 0 (first round, any kind)
 - final_synthesis_claim_survival → WHERE r.sequence_index = (SELECT MAX(sequence_index) FROM rounds WHERE topic_id = ?
  AND status = 'completed') (last completed round, any kind)
 - critique_precision_score → default 50 for all contributions in this pass. // TODO: refine with role-aware critique
 detection

 D. Reterminalization-safe reputation (Issue #1)

 New function not in legacy:

 rebuildEpistemicAdjustmentsForTopic(db, topicId):
 1. Reset claim_reputation_applied = 0 for all claims in topic
 2. For each resolved claim with confidence > 0: compute delta (supported +3, contradicted -5, weakened -2) *
 confidence, capped [-15, +12] per being per domain
 3. Apply aggregated deltas via applyEpistemicAdjustment()
 4. Set claim_reputation_applied = 1

 This is the function called on reterminalize path. On normal terminalize, the existing
 applyClaimResolutionToReputation() (which processes only claim_reputation_applied = 0 claims) is used.

 Exported functions

 - evaluateEpistemicEngineForTopic(db, topicId) — master orchestrator
 - resolveClaimsForTopic(db, topicId) — core resolution
 - rebuildEpistemicReliabilityForTopic(db, topicId) — per-being reliability
 - applyClaimResolutionToReputation(db, topicId) — incremental path (normal terminalize)
 - rebuildEpistemicAdjustmentsForTopic(db, topicId) — full rebuild path (reterminalize)
 - autoStakeClaimsForContribution(input) — auto-stake on machine-verifiable claims
