# Phase 5 Plan: Closure, Reveal Enforcement, Reputation Repair, and Public Output Correctness

  ## Summary

  Implement Phase 5 as the closure layer on top of the current Phases 1-4 runtime. The scope is: completion-style-
  aware round closure, reveal-gated visibility, idempotent lifecycle progression, repair-safe reputation
  recomputation, terminalization-quality-based artifacts, explicit presentation retry, and cache invalidation signals.

  This plan keeps Phase 1’s shared-contract rule intact by defining new closure, invalidation, artifact, and repair
  contracts in packages/shared, then wiring Phase 2 lifecycle, Phase 3 visibility/scoring reads, and Phase 4 final-
  score semantics to those contracts.

  ## Implementation Order

  1. Add shared contracts, enums, constants, and schemas in packages/shared.
  2. Implement Phase 2 lifecycle upgrades:
     idempotent CAS advancement, reveal_at initialization, early-completion timing rewrites.
  3. Add reveal-gated visibility helpers and update all Phase 3/4 read paths to use them.
  4. Upgrade terminalization and repair-safe reputation recomputation.
  5. Split snapshot sync from presentation reconcile, then add presentation retry plumbing.
  6. Implement verdict HTML and OG SVG artifact generation and publication/suppression logic.
  7. Update internal repair routes and /meta/contract to publish the Phase 5 contract from shared definitions.
  8. Add and run the full test matrix.

  ## Implementation Changes

  ### 1. Round timing, reveal_at, and idempotent progression

  - Define a round as “sealed” when round_configs.config_json.visibility === 'sealed'.
  - At topic creation, populate every round’s initial reveal_at from its planned timing:
      - sealed round: reveal_at = ends_at
      - open round: reveal_at = starts_at
  - On actual completion of a round:
      - set that round’s stored ends_at to the actual completion timestamp
      - set that round’s reveal_at to the actual completion timestamp
  - When early completion shifts downstream timing:
      - set next round starts_at = now
      - recompute downstream starts_at, ends_at, and planned reveal_at by preserving each round’s original duration
        from topic creation
      - for pending downstream rounds:
          - sealed round: reveal_at = recomputed ends_at
          - open round: reveal_at = recomputed starts_at
      - for already-completed rounds, do not rewrite historical reveal_at
  - Replace deadline-only advancement with a compare-and-set progression service using D1 conditional updates:
      - UPDATE rounds SET status = 'completed', ends_at = ?, reveal_at = ? WHERE id = ? AND status = 'active'
      - UPDATE rounds SET status = 'active', starts_at = ? WHERE id = ? AND status = 'pending'
      - UPDATE topics SET status = 'closed', closed_at = ? WHERE id = ? AND status != 'closed'
  - Treat D1’s meta.changes as the CAS success signal.
  - Duplicate sweeps must no-op if meta.changes === 0.
  - Preserve Phase 2 runtime assumptions:
      - enrollment remains open topic-member runtime only
      - current_round_index remains the active round’s sequence index
      - topic status remains started until terminal closure or stalled

  ### 2. Completion-style execution

  - Add a round completion evaluator service that reads:
      - round_configs.config_json
      - active topic members
      - transcript-visible contributions
      - contribution scores
  - Execute completion styles as:
      - aggressive: complete when contribution count > 0 and distinct contributors meet
        DEFAULT_TOPIC_MIN_DISTINCT_PARTICIPANTS; otherwise deadline fallback
      - patient: complete when all active members contributed, otherwise deadline fallback
      - quality_gated: require minimum elapsed duration and quality threshold, otherwise deadline fallback
  - quality_gated threshold:
      - max(QUALITY_GATED_MIN_SCORE_FLOOR, median(final_score) of transcript-visible prior-round contributions)
      - if no usable prior-round final-score set exists, use QUALITY_GATED_MIN_SCORE_FLOOR
  - Completion evaluator and advancement service remain separate:
      - evaluator decides whether a round is eligible to close
      - advancement service performs CAS-style mutations and downstream timing rewrites

  ### 3. Reveal enforcement across Phase 3 and Phase 4 read paths

  - Phase 5 makes reveal_at authoritative for read visibility, not just metadata.
  - Add a shared API helper for “transcript-visible contribution” selection:
      - contribution must have allowed transcript visibility state
      - if the round is sealed, it is transcript-visible only when reveal_at <= now
      - if the round is open, reveal gating does not hide it
  - Apply reveal-gated filtering to:
      - snapshot-sync.ts
      - contribution scoring context reads in routes/contributions.ts
      - vote target lookup in services/votes.ts
  - Keep Phase 3 ordering invariant:
      - transcript snapshot first
      - state snapshot second
  - State snapshots may include round metadata for unrevealed rounds, but contribution bodies/scores from sealed-
    unrevealed rounds must not appear in transcript-visible collections.

  ### 4. Terminalization, force-flush protocol, and Phase 4 score semantics

  - Keep runTerminalizationSequence() as the authoritative closeout entrypoint.
  - Force-flush protocol:
      - call the DO /force-flush endpoint
      - endpoint flushes pending contributions, scores, and votes once, then returns { flushed: boolean, remaining:
        number }
      - terminalization repeats until remaining === TERMINALIZATION_FORCE_FLUSH_EMPTY_REMAINING or attempts reach
        TERMINALIZATION_FORCE_FLUSH_MAX_ATTEMPTS
      - if still not drained after the cap, fail terminalization and queue presentation/terminalization retry rather
        than proceeding on partial state
  - Terminalization order:
      1. force-flush until drained or capped failure
      2. reload authoritative topic/round/contribution/vote state
      3. short-circuit if topic is not closed
      4. short-circuit if verdict exists unless explicit reterminalize mode
      5. recompute canonical final scores for all topic contributions from authoritative votes
      6. recompute or rebuild affected domain reputation state
      7. determine terminalization mode and confidence
      8. write verdict
      9. run presentation reconcile
  - Explicit Phase 4 tie-in:
      - reterminalization recomputes only canonical final_score and shadow_final_score from authoritative votes
      - initial_score and shadow_initial_score remain immutable ingest-time values
      - live_score remains the compatibility mirror of initial_score
      - shadow_score remains the compatibility mirror of shadow_initial_score
      - score version fields preserve existing semantics and are not repurposed during repair

  ### 5. Reputation update and initial-score boost

  - Confirm and preserve the existing Phase 4 behavior where domain reputation boosts initial score during ingest-time
    composite scoring, capped at 20%.
  - Phase 5 does not move that boost; it depends on it remaining in the contribution ingest path.
  - Reputation writes still occur only during terminalization and repair.
  - For normal terminalization:
      - append per-contribution final scores into domain_reputation using current Welford update logic
  - For reterminalize:
      - identify affected (domain_id, being_id) pairs from the topic’s closed contributions
      - for each pair, query all authoritative contributions from closed topics in that domain for that being, ordered
        deterministically
      - rebuild Welford aggregate from scratch
      - overwrite the corresponding domain_reputation row
      - then reapply decay from last_active_at using existing rules
  - Repair mode must not append duplicate samples.

  ### 6. Snapshot vs presentation reconcile call-site migration

  - Split current behavior into:
      - syncTopicSnapshots() for transcript/state/curated-open JSON only
      - reconcileTopicPresentation() for snapshot sync + artifact publication/suppression + cache invalidation + retry
        bookkeeping
  - Caller migration rules:
      - DO flush:
          - stays snapshot-only for non-terminal contribution/vote activity
          - must not trigger full presentation reconcile on every hot-path flush
      - scheduled snapshot retry in index.ts:
          - remains snapshot-only and drains only snapshot-pending:*
      - lifecycle sweep:
          - for ordinary round/topic mutations before closure, keep snapshot-only
          - when a topic becomes closed, call terminalization, and terminalization calls full presentation reconcile
      - internal repair routes:
          - reconcile-presentation calls full presentation reconcile only
          - reterminalize calls full terminalization then full presentation reconcile
  - This preserves Phase 3 snapshot behavior while adding Phase 5 closeout behavior only where closure or repair
    requires it.

  ### 7. Presentation failure and retry flow

  - Add distinct presentation retry plumbing alongside snapshot retry.
  - Queue location:
      - KV under presentation-pending:{topicId}
  - Payload:
      - topicId
      - reason
      - queuedAt
      - attemptCount
      - last error class/message summary
  - On any failure inside reconcileTopicPresentation():
      - set topic_artifacts.artifact_status = 'error'
      - write/update the presentation retry KV item
      - do not roll back already-written transcript/state snapshots
  - Scheduler behavior in index.ts:
      - before lifecycle sweep, drain both retry queues
      - snapshot retry queue calls syncTopicSnapshots()
      - presentation retry queue calls reconcileTopicPresentation()
  - Successful presentation reconcile:
      - deletes presentation-pending:{topicId}
      - updates topic_artifacts to published or suppressed as appropriate

  ### 8. Artifact generation contract

  - Artifact generation is explicit and Phase-5-bounded.
  - Inputs:
      - topic state snapshot payload
      - verdict row
      - top contributions per completed round from verdict reasoning JSON
      - terminalization mode and confidence
  - Outputs:
      - verdict HTML rendered from a deterministic server-side template in the API package
      - OG SVG rendered from the same structured data as a static share asset
  - Storage rules:
      - verdict HTML: artifacts/topics/{topicId}/verdict.html, content type text/html; charset=utf-8
      - OG SVG: artifacts/topics/{topicId}/og.svg, content type image/svg+xml
  - Overwrite behavior:
      - publication is upsert-style; reruns overwrite the same keys
      - suppression deletes both keys if present and nulls verdict_html_key / og_image_key
  - Artifact status contract:
      - pending: metadata row exists but no snapshots yet
      - ready: transcript/state snapshots are current but no closed-topic artifacts published
      - published: verdict HTML and OG SVG exist for a closure strong enough to publish
      - suppressed: closure is insufficient signal, so closed-topic artifacts are intentionally absent
      - error: reconcile failed and retry is queued
  - Artifact policy:
      - full_template and degraded_template publish both artifacts and set artifact_status = 'published'
      - insufficient_signal removes both artifacts and sets artifact_status = 'suppressed'
  - Transcript/state snapshots always remain authoritative regardless of artifact suppression.

  ### 9. Cache invalidation contract

  - Add an API-side invalidation service with shared key names.
  - Invalidation operations:
      - bump KV generation counters for landing/domain/topic/verdict surfaces
      - refresh curated/open.json
      - record invalidation reason and timestamp in KV for repair/debugging
  - Trigger invalidation from:
      - round completion that changes visible transcript/state
      - topic closure
      - successful terminalization
      - successful presentation reconcile repair
      - successful reterminalize repair
  - Phase 5 publishes invalidation signals only; router consumption remains Phase 6.

  ## Non-Goals During Execution

  - No selective enrollment or fallback-chain runtime work.
  - No router rendering work beyond producing artifacts and invalidation signals that Phase 6 can consume.
  - No new DB tables, no new score columns, and no changes to Phase 4 score-version semantics.
  - No PNG raster pipeline or external image-rendering service; OG assets ship as SVG in Phase 5.
  - No admin dashboard expansion beyond the two repair endpoints required by this plan.

  ## Test Plan

  - Round creation tests:
      - every round receives initial starts_at, ends_at, and non-null reveal_at
      - sealed rounds initialize reveal_at = ends_at
      - open rounds initialize reveal_at = starts_at
  - Timing rewrite tests:
      - early completion clamps current round ends_at
      - current round reveal_at becomes actual completion time
      - next round starts_at becomes actual activation time
      - downstream starts_at/ends_at/planned reveal_at are recomputed from preserved durations
      - recomputed sealed rounds use reveal_at = recomputed ends_at
      - recomputed open rounds use reveal_at = recomputed starts_at
      - terminal close sets actual topics.closed_at
  - Idempotent advancement tests:
      - duplicate sweeps cannot complete the same round twice
      - duplicate sweeps cannot activate the same next round twice
      - duplicate sweeps cannot close the same topic twice
      - D1 meta.changes === 0 is treated as a no-op path
  - Completion-style tests:
      - aggressive early completion
      - patient all-members completion
      - quality_gated threshold and minimum-duration enforcement
  - Reveal enforcement tests:
      - sealed-active contributions remain hidden before ends_at
      - sealed-unrevealed contributions excluded from transcript snapshots
      - sealed-unrevealed contributions excluded from scoring context
      - vote target lookup respects reveal-gated transcript visibility rules
  - Force-flush tests:
      - DO endpoint returns { flushed, remaining }
      - terminalization stops when remaining === 0
      - terminalization aborts after TERMINALIZATION_FORCE_FLUSH_MAX_ATTEMPTS
  - Terminalization tests:
      - score recomputation updates only final-score columns
      - normal rerun is idempotent
      - repair mode replaces verdict/presentation safely
  - Reputation tests:
      - ingest path still applies the capped initial-score reputation boost
      - normal terminalization appends Welford samples once
      - reterminalize performs authoritative aggregate rebuild for affected pairs
      - rebuild does not double-count topic samples
      - decay still respects grace window and floor
  - Artifact tests:
      - correct key naming and content types
      - verdict HTML renderer consumes structured verdict/topic inputs
      - OG SVG renderer consumes the same canonical input set
      - publish overwrites stable keys
  - Presentation retry tests:
      - presentation retry KV record is created
      - scheduler drains presentation retry queue successfully
  - Cache invalidation tests:
      - expected generation counters bump on closure and repair
      - invalidation reason/timestamp KV entries are written

  ## Assumptions and Defaults

  - No new tables are added in Phase 5.
  - Early completion preserves original round durations rather than inventing new template timing rules.
  - “Sealed round” means round_configs.config_json.visibility === 'sealed'.
  - Sealed rounds initialize reveal_at = ends_at; open rounds initialize reveal_at = starts_at.
  - Reveal gating is authoritative for transcript visibility, even when round metadata is otherwise visible.
  - Repair-mode reputation uses authoritative full rebuild for affected (domain_id, being_id) aggregates because no
    topic-scoped reputation ledger exists.
  - Hot-path DO flush remains snapshot-only outside closure/repair paths to preserve Phase 3 behavior and avoid
    unnecessary artifact churn.
  - The existing ingest path already applies the required reputation boost on initial score and Phase 5 preserves that
    contract rather than relocating it.
  - Phase 5 publishes OG assets as SVG, not PNG, to avoid introducing ambiguous rendering dependencies in the Worker
    runtime.
